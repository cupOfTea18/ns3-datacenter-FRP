# Flow 场景设计分析

> 本文档分析 `examples/PowerTCP/flow copy.txt` 中设计的 5 个测试场景，说明每个场景的瓶颈位置、对应的监控端口，以及 `run_single_algo.py` 需要做的相应调整。

---

## 1. 拓扑与 DC 划分

### 1.1 拓扑总览（来自 `topology.txt`）

```
14 6 4 13
8 9 10 11 12 13
```

| Host | 接入 ToR | 所属 DC |
|------|---------|---------|
| H0, H1 | ToR8  | **DC1** |
| H2, H3 | ToR9  | **DC1** |
| H4, H5 | ToR10 | **DC2** |
| H6, H7 | ToR11 | **DC2** |

| 交换机 | 角色 | DC |
|--------|------|----|
| 8  | ToR   | DC1 |
| 9  | ToR   | DC1 |
| 12 | Spine | DC1 |
| 10 | ToR   | DC2 |
| 11 | ToR   | DC2 |
| 13 | Spine | DC2 |

### 1.2 拓扑结构图

```
       ┌────────── DC1 ──────────┐               ┌────────── DC2 ──────────┐
       │                          │   跨DC链路    │                          │
       │       Spine12  ◀────────────────────────▶  Spine13                 │
       │         △                │ 200G, 1.1ms  │     △                    │
       │         │                │              │     │                    │
       │   ┌─────┴─────┐          │              │   ┌─┴────────┐           │
       │   │  100G     │          │              │   │   100G    │          │
       │   │  1.5μs    │          │              │   │   1.5μs   │          │
       │   ▼           ▼          │              │   ▼           ▼          │
       │  ToR8        ToR9        │              │  ToR10       ToR11       │
       │  △△          △△          │              │  △△          △△          │
       │  ││          ││          │              │  ││          ││          │
       │  ▼▼          ▼▼          │              │  ▼▼          ▼▼          │
       │ H0 H1       H2 H3        │              │ H4 H5       H6 H7        │
       └──────────────────────────┘              └──────────────────────────┘
```

### 1.3 跨 DC 流量判断

| 流向 | DC 关系 | 是否跨 DC | 必经路径 |
|------|---------|-----------|---------|
| H0/H1/H2/H3 ↔ H0/H1/H2/H3 | DC1 内 | ❌ 否 | 不经 Spine12 ↔ Spine13 |
| H4/H5/H6/H7 ↔ H4/H5/H6/H7 | DC2 内 | ❌ 否 | 不经 Spine12 ↔ Spine13 |
| H0~H3 ↔ H4~H7 | DC1 ↔ DC2 | ✅ **跨 DC** | 必经 Spine12 ↔ Spine13 |

## 2. 监控点说明

### 2.1 Switch 10（ToR10）端口映射

按 `topology.txt` 链路顺序，ns-3 中 Switch 10 的端口编号：

| Port | 对端 | 方向 | 含义 |
|------|------|------|------|
| 0 | loopback | - | ns-3 默认 |
| **1** | **Host 4** | 下行（→ H4） | **远端拥塞监控点** |
| 2 | Host 5 | 下行（→ H5）| - |
| **3** | **Spine 13** | 上行（→ 跨DC方向）| **近端拥塞监控点** |

### 2.2 Python 监控逻辑（`run_single_algo.py`）

```python
# DCQCN/HPCC/TIMELY: 匹配 [DCQCN_QLEN] switch=10, port=?
# FRP/ROCC: 匹配 [FRP_DATA_SW] switch=10, port=?
if int(parts[2]) == 10 and int(parts[3]) == X:
```

**根据测试场景，把 X 改成对应的端口号：**

| 场景类型 | `parts[3] ==` |
|---------|--------------|
| 远端拥塞 | **1** |
| 近端拥塞 | **3** |

## 3. 5 个场景逐个分析

### 场景 1：一条跨域流 + 一条 DC 内流，**远端拥塞**

```
2
0 4 3 10031 10000000000 0    # Flow1: H0(DC1) → H4(DC2)  跨DC
5 4 3 10032 10000000000 0    # Flow2: H5(DC2) → H4(DC2)  DC2内（同ToR）
```

#### 流路径

| Flow | 路径 | 类型 |
|------|------|------|
| H0 → H4 | H0 → ToR8 → Spine12 → **Spine13** → ToR10 → H4 | 跨 DC |
| H5 → H4 | H5 → ToR10 → H4 | DC2 内同 ToR |

#### 瓶颈分析

| 项 | 值 |
|----|----|
| 瓶颈位置 | **Switch 10 Port 1**（→ H4，100Gbps）|
| 入流总量 | 200Gbps（远超 100Gbps 出口） |
| Python 监控配置 | `parts[3] == 1` ✅（保持默认）|

✅ **场景 1 可直接运行**

### 场景 2：两条跨域流 + 两条 DC 内流，**远端拥塞**

```
4
0 4 3 10031 10000000000 0    # Flow1: H0(DC1) → H4(DC2)  跨DC
1 4 3 10032 10000000000 0    # Flow2: H1(DC1) → H4(DC2)  跨DC
5 4 3 10033 10000000000 0    # Flow3: H5(DC2) → H4(DC2)  DC2内同ToR
6 4 3 10034 10000000000 0    # Flow4: H6(DC2) → H4(DC2)  DC2内跨ToR
```

#### 流路径

| Flow | 路径 | 类型 |
|------|------|------|
| H0 → H4 | H0 → ToR8 → Spine12 → **Spine13** → ToR10 → H4 | 跨 DC |
| H1 → H4 | H1 → ToR8 → Spine12 → **Spine13** → ToR10 → H4 | 跨 DC |
| H5 → H4 | H5 → ToR10 → H4 | DC2 内同 ToR |
| H6 → H4 | H6 → ToR11 → **Spine13** → ToR10 → H4 | DC2 内跨 ToR |

#### 瓶颈分析

| 项 | 值 |
|----|----|
| **主瓶颈** | **Switch 10 Port 1**（→ H4，100Gbps），4 条流汇聚 ≈ 400Gbps |
| 次级瓶颈 | **Spine13 → ToR10** 链路：3 条流（Flow1+2+4）≈ 300Gbps > 100Gbps |
| Python 监控配置 | `parts[3] == 1` ✅（保持默认）|

✅ **场景 2 可直接运行**（比场景 1 拥塞更严重）

#### 🟡 附加说明
Spine13 → ToR10 这条链路也会拥塞，但 Python 不监控这里。如果想看次级瓶颈，需要在 Switch 13 上额外打日志或扩展监控范围。

### 场景 3：一条跨域流 + 一条 DC 内流，**近端拥塞**

```
2
4 0 3 10031 10000000000 0    # Flow1: H4(DC2) → H0(DC1)  跨DC，源在H4
5 1 3 10032 10000000000 0    # Flow2: H5(DC2) → H1(DC1)  跨DC，源在H5
```

#### 流路径

| Flow | 路径 | 经过 ToR10 端口 |
|------|------|-----|
| H4 → H0 | H4 → **ToR10** → **Spine13** → Spine12 → ToR8 → H0 | Port 1(入) → Port 3(出) |
| H5 → H1 | H5 → **ToR10** → **Spine13** → Spine12 → ToR8 → H1 | Port 2(入) → Port 3(出) |

#### 瓶颈分析

| 项 | 值 |
|----|----|
| **瓶颈位置** | **Switch 10 Port 3**（ToR10 → Spine13，100Gbps）|
| 入流总量 | 200Gbps（H4 + H5 各 100Gbps） |
| 出口 | 100Gbps（ToR10 → Spine13）|
| Python 监控配置 | **`parts[3] == 3`** ⚠️ 需修改 |

✅ **场景 3 设计正确**：两条流都从 DC2 主机出发，争抢 ToR10 → Spine13 这条上行链路

#### 🛠️ 运行前需要修改 `run_single_algo.py`

在第 99 行和第 106 行：

```python
# DCQCN/HPCC/TIMELY
if len(parts) >= 5 and int(parts[2]) == 10 and int(parts[3]) == 1:
                                                              ^^^
                                                          改为 3

# FRP/ROCC
if len(parts) >= 8 and int(parts[2]) == 10 and int(parts[3]) == 1:
                                                              ^^^
                                                          改为 3
```

> ⚠️ 注意：`flow copy.txt` 中场景 3 的描述写的是"一条跨域流和一条DC内流"，但实际两条流（H4→H0 和 H5→H1）**都是跨 DC 流**。建议把描述改为"两条跨域流在近端拥塞"，或者把 Flow2 改成 `5 6 3 10032 ...`（H5→H6 是 DC2 内跨 ToR，也走 ToR10 上行）。

### 场景 4：两条跨域流 + 两条 DC 内流，**近端拥塞**

```
4
4 0 3 10031 10000000000 0    # Flow1: H4 → H0 (跨DC)
5 1 3 10032 10000000000 0    # Flow2: H5 → H1 (跨DC)
4 6 3 10033 10000000000 0    # Flow3: H4 → H6 (DC2内跨ToR)
5 7 3 10034 10000000000 0    # Flow4: H5 → H7 (DC2内跨ToR)
```

#### 流路径

| Flow | 路径 | 经过 ToR10 端口 |
|------|------|-----|
| H4 → H0 | H4 → ToR10 → **Spine13** → Spine12 → ToR8 → H0 | Port 1(入) → Port 3(出) |
| H5 → H1 | H5 → ToR10 → **Spine13** → Spine12 → ToR8 → H1 | Port 2(入) → Port 3(出) |
| H4 → H6 | H4 → ToR10 → **Spine13** → ToR11 → H6 | Port 1(入) → Port 3(出) |
| H5 → H7 | H5 → ToR10 → **Spine13** → ToR11 → H7 | Port 2(入) → Port 3(出) |

#### 瓶颈分析

| 项 | 值 |
|----|----|
| **瓶颈位置** | **Switch 10 Port 3**（ToR10 → Spine13，100Gbps）|
| 入流总量 | 400Gbps（H4 200G + H5 200G）|
| 出口 | 100Gbps |
| Python 监控配置 | **`parts[3] == 3`** ⚠️ 需修改 |

✅ **场景 4 设计正确**：4 条流全部从 ToR10 上行，比场景 3 拥塞更严重

#### 🟢 设计亮点
- 4 条流都不进入 H4（H4 既不是接收方），所以 **Switch 10 Port 1 不会拥塞**——拥塞完全集中在 Port 3
- 这是纯粹的"近端拥塞"场景，没有混合远端拥塞

#### ⚠️ NIC 限速注意
H4 和 H5 各自承担 2 条流（每条 100Gbps），但单 NIC 只有 100Gbps，所以**每台主机实际上行最多 100Gbps**，总入流被限制在 200Gbps（不是 400Gbps）。但仍然 > 100Gbps 出口，会拥塞。

### 场景 5：FCT 测试（4 条背景流 + 10 条小流）

```
14
0 4 3 10001 10000000000 0      # 大流1: H0→H4 (跨DC, 10GB)
1 4 3 10002 10000000000 0      # 大流2: H1→H4 (跨DC, 10GB)
5 4 3 10003 10000000000 0      # 大流3: H5→H4 (DC2内, 10GB)
6 4 3 10004 10000000000 0      # 大流4: H6→H4 (DC2内, 10GB)
2 4 3 10005 10000 0.001        # 小流1: H2→H4 (10KB, 1ms启动)
3 4 3 10006 10000 0.002        # 小流2: H3→H4 (10KB, 2ms启动)
7 4 3 10007 10000 0.003        # 小流3: H7→H4 (10KB, 3ms启动)
2 4 3 10008 10000 0.005        # 小流4
3 4 3 10009 10000 0.008        # 小流5
7 4 3 10010 10000 0.010        # 小流6
2 4 3 10011 10000 0.015        # 小流7
3 4 3 10012 10000 0.020        # 小流8
7 4 3 10013 10000 0.025        # 小流9
2 4 3 10014 10000 0.030        # 小流10
```

#### 设计分析

| 角色 | 流 | 大小 | 源 | 目的 |
|-----|----|----|----|----|
| 背景大流 | Flow 1-4 | 10GB | H0/H1/H5/H6 | H4 |
| 测试小流 | Flow 5-14 | 10KB | H2/H3/H7（与大流源不重叠）| H4 |

#### 路径与瓶颈

- 所有 14 条流的目的都是 **H4**
- 所有流最终汇聚到 **Switch 10 Port 1**（→ H4）
- 4 条 10GB 大流持续打满瓶颈，10 条 10KB 小流穿插

#### 瓶颈分析

| 项 | 值 |
|----|----|
| 瓶颈位置 | **Switch 10 Port 1**（→ H4，100Gbps）|
| Python 监控配置 | `parts[3] == 1` ✅（保持默认）|

#### 🟡 配套修改建议

1. **延长仿真时长**（`run_single_algo.py:20`）：
   ```python
   DURATION = 0.100   # 100ms，让最晚启动的小流（30ms 启动）有足够时间完成
   ```
   - 原来的 20ms 太短，30ms 启动的小流根本不会运行

2. **FCT 数据查看路径**：
   ```
   simulator/ns-3.39/mix/fct.txt
   ```
   每行格式：`sip dip sport dport size start_time(ns) fct(ns) standalone_fct(ns)`

3. **关键评估指标**：
   - **小流 FCT**：理想值 ≈ 10KB / 100Gbps + RTT ≈ 几 μs
   - **slowdown** = 实际 FCT / standalone FCT，越接近 1 越好
   - 跨 DC 小流（H2/H3 → H4）和 DC2 内小流（H7 → H4）的 FCT 应该有显著差异

#### ⚠️ 注意

- 大流大小写成 `10000000000`（10GB），单条流就需要 10GB × 8 / 100Gbps = **800ms** 才能跑完。即使 100ms 仿真也只跑 12.5%。
- 这对 FCT 测试**没有影响**（小流能完成就行），但要注意 `fct.txt` 里**不会有大流的记录**（大流没完成）。
- 如果想让大流也能完成，可以把大小改小（比如 `100000000` = 100MB，每条 8ms 即可完成）。

## 6. 推荐：把监控端口改成可配置参数

避免每次切换场景都要改两处代码，建议在 `run_single_algo.py` 顶部加一个全局参数：

```python
NS3_DIR = "/home/shemuping/newCode/ns3-FRP/simulator/ns-3.39"
RESULTS_DIR = "/home/shemuping/newCode/ns3-FRP/results"
CONFIG_FILE = "examples/PowerTCP/config.txt"
DURATION = 0.020

# === 监控配置 ===
MONITOR_SWITCH = 10
MONITOR_PORT = 1     # 远端拥塞场景(1,2,5) → 1；近端拥塞场景(3,4) → 3
```

然后日志解析处用变量：
```python
if int(parts[2]) == MONITOR_SWITCH and int(parts[3]) == MONITOR_PORT:
    ...
```

切换场景时只改这一个变量即可。

## 7. 快速排错清单

| 现象 | 可能原因 | 检查方法 |
|------|---------|---------|
| 队列曲线全是 0 | 监控端口选错（如远端场景选了 Port 3）| 对照本文档 §4 表格 |
| 速率曲线起点不是 0 | 已修复（rdma-hw.cc 中的 qpKey）| 确认已重新编译 |
| FCT 数据为空 | DURATION 太短，流没完成 | 加大 DURATION，或减小 flow size |
| 看不到所有 Host 的速率 | `[TX RATE]` 采样 50 包/次，流刚启动可能没打印 | 加大流大小或降低采样间隔 |
